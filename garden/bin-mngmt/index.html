Managing Bulky Binary Assets

Large binary files are a cost on any substantially sized project that requires them. Microsoft's Vcpkg manager presents an interesting possible solution.

During software development, being able to view of a project’s history across time has many positive implications for compatibility, security, and just general learning. Because of the very nebulous nature of solving this problem and a relative necessity for project creators to solve this problem, there are a number solution platforms availible. Of these different kinds of solutions in the set, Git versioning is pretty ubiquitously considered the best local minimum across the problem domain.

Some Motivating Problems#

One of the biggest points of friction for Git occurs when trying to integrate external assets into a project. This is problem rears its ugly head the most when it comes to game projects and any project that requires lots of art assets. This can come in the form of precompiled binary files, image files, databases, 3d models, and more. These file types are bulky, bloated, and are hard for Git to compare differences to.

Git only saves the differences between multiple versions of the same file as opposed to the file in it’s entirety multiple times to prevent from tracking redundant information. This means that a single file could be edited a huge number of times but still keep the .git/ folder that saves metadata for the project as lightweight as possible. The problem with binary files occurs due to the nature of how most binary files are created. Regardless of the data being saved, most often a small change in the data the binary file represents can make a massive change in the underlying structure of the binary. This means that tracking all of the possible information that the binary file can have results in a larger memory footprint inside of Git, and in turn gives the repository a larger memory footprint.

Another corollary of storing assets directly in the version control system (VCS) is that it requires artists who may or may not have any kind of real technical prowess to interact with the system. This creates a pretty heavy knowledge floor that in the best case requires the hiring of artists who already have knowledge of the underlying VCS technology. This could potentially lead to significant decrease in the set of talent that can be hired, and in the worst case could cause an artist who already has experience with the system to sink unnecessary time in fighting the version control versus actually practicing their profession.

Odin: A Memory Footprint Case Study#

A working example of this can be seen in the repository for the Odin programming language ↗. The language repository commits external libraries, such as SDL3 ↗ and raylib ↗ directly into the source tree so that maintainers can have easy access to any necessary dependencies during development.

The direct inclusion of vendored libraries provides a convenience for anyone who wants to get started either in development on the language or creating a new project using the language. This also follows closely with Odin’s design philosophy for being simple with a focus on actually creating software projects instead of focusing on the framework supporting them.

After cloning the repository with a given SHA hash for consitency ↗ and no further manipluation, Windows calculates the total size of the repository as 930 MB and the total size of the .git/ metadata folder as 520 MB. This is a percentage footprint of over 50%!

We can further break this down to do an analysis on the Git metadata itself to see where most of this memory is accumulating. After running the following shell commands from inside the repository we can get an output of the different file sizes of each commit.

git rev-list --objects --all --missing=print |
  git cat-file --batch-check='%(objecttype) %(objectname) %(objectsize) %(rest)' |
  sed -n 's/^blob //p' |
  sort --numeric-sort --key=2 |
  cut -c 1-12,41- |
  $(command -v gnumfmt || echo numfmt) --field=2 --to=iec-i --suffix=B --padding=7 --round=nearest
bash

Putting aside the fact that this set of shell commands looks like pure arcane gibberish to anyone who lacks a pretty intimate knowledge of Git, the resulting output tells an interesting story. Virtually all of the files that appear in the list that are greater than 1 MiB (outside of a few generated C source and header files) are bulky binary files. You see files like bin/lld-link.exe, wgpu_native.lib, and in particular LLVM-C.dll, which are not only greater than 100 MiB each but also appear multiple times in the repository’s history. LLVM-C.dll alone comprises of 719 MiB in total memory footprint. Compare this to a source file like core/sys/windows/types.odin which appears drastically more times, 155 times to be exact, in the repository’s history and only has a memory footprint of 14.126 MiB. It’s important to note that these numbers don’t reflect their size inside of the .git/ folder, but rather the total amount of information ↗ necessary in total to encode them.

This very short case study is not to say that this approach is incorrect in the slightest, Odin is an amazing language that accomplishes its goals very well. However, it is more to ask the question; what is the cost of this convenience? Moreover, it’s to highlight the fact that Git is a solution addressing the source control problem that lives in a local minimum.

Supply Chain Attacks#

Another weakness of direct inclusion appears in the security domain. This practice increases the probability that your repository is vulnerable to supply chain attacks ↗. I’m no security expert, so I won’t pretend to speak with any kind of authority on the matter, but the inclusion of binary files in a Git repository appears like it would increase the attack surface of the project. I’d also hypothesize that this potential vulnerability is not limited to when your project has external contributors. Files types that are used in asset production, such as Blender’s .blend files, are binary files that sometimes require the use of 3rd party addons and plugins that an artist innocently uses to produce their work. These addons could possibly contain executable code that, when used, poison the project in conjunction with Git and GitHub’s automation functionality.

Such vulnerabilties within the scope of my limited knowledge are strictly theoretical. None the less, even the possiblity of their existence in an attack vector raises questions about the safety directly including binary files in a repository.

Where To Go From Here?#

Git has a few different workarounds for this problem, one of which that has been developed relatively recently (development began sometime around 2014) is Git LFS ↗. LFS attempts to track the checksums of each file type the maintainer determines is adequate, along with some other metadata. This results in a hugely reduced memory footprint since that metadata is all stored in a plaintext format, and the large files exist solely on the server until required. While this extension to Git provides some interesting ideas, it still has one major flaw. Artists still have to interact with the VCS directly.

To address the problem of vendoring external libraries for C/C++ projects, Microsoft created the package management tool vcpkg ↗. Vcpkg provides an interface for client users to fetch, build, and vendor 3rd party libraries through two simple configuration files and a relatively digestable CLI. This interface also supports vendoring libraries on multiple platforms, so client users can aqcuire the pacakges they need regardless of the platform or CPU architecture they are operating on. Vcpkg’s core tenents are about improving the C/C++ build pipeline to make any pipeline better at supporting cross-platform compilation as well as adding external binary libraries to any possible build and project system. Taken directly from the project’s overview page in the official documentation, one of its explicit goals is to “Build dependencies from source or download prebuilt ABI-verified binaries, with over 70 configurations available by default, and infinite customization for your specific requirements”.

At this point, you might be confused as to how a C/C++ build tool solves, in part or in full, the bulky binary file sub-problem of version control. The answer also comes from the vcpkg homepage ↗ as of the time of writing, allowing users to “add your own private libraries to simplify your build process”. Precompiled C/C++ libraries, like the ones that vcpkg is designed to support by default, are just binary files. Should custom vcpkg registries provide a configurable enough interface, we should be enabled to vendor any arbitrary binary file. As it turns out, this is the case. Inside our project’s vcpkg_configuration.json, all we need to do is specify a reference to where the registry exists and vcpkg takes care of the rest. Even more importantly, this reference can be a registry that is not publicly available ↗ and we can still utilize vcpkg’s functionality.

The custom registry itself comprises of configuration code that instructs vcpkg of what to do when a client tries to vendor a specifc “port”, where a port is a collection of binary assets the client wants to use. Client users and companies can define a port for a variety of use cases. It can be a character asset collection, pre-compiled libraries, texture libraries, pre-compiled shaders, and more. All the client user needs to do is implemnt the portfile.cmake and vcpkg.json configurations for their custom port and vcpkg will take care of the rest when a developer invokes vcpkg install from inside the project.

Crucially, ports can reference almost any type of endpoint. That endpoint can be a server where all an artist needs to do is upload their work and not even need to have a single line of code on their computer to be able to keep contributing to the project. Custom registries can also define any amount of ports, allowing a single registry to be used across projects, providing a company or team with very granular reusablity.

Some Closing Remarks#

It should be important to note that this is not an advertisement for vcpkg. There are a bunch of glaring problems with using it in this way, or even using it in general. Most notably, the fact that its an open source Microsoft project, a company that famously has subscribed to the motto of “Embrace, extend, and extinguish” ↗ in the past. Additionally, the heart of registry configuration is done in CMake. Even though Microsoft provides examples for vcpkg ports that don’t use CMake ↗ as their build system, anyone who has done any kind of C/C++ development using CMake should be intimately aware of why this could be a problem.

Instead of purely advocating for the use of vcpkg as the solution, this train of thought is more so designed to think about a way to solve this bulky binary management sub-problem of Git. It is also attempting to avoid the work of writing a custom VCS from scratch and instead use Git as a bootstrap towards a more optimal minimum.
